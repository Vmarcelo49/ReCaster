feat(relay): validate room code before handshake + richer GUI status

== Problem ==
When joining via relay room code (#ABCD), the user had to wait up to 60s
(MatchInfoTimeout) to discover the room didn't exist. The GUI showed
only generic "Connecting to relay server..." with no indication of what
was happening or why it was failing.

== Fix 1: One-shot room code validation (relay_client) ==
Added `validate_room_code()` to relay_client — a blocking TCP probe
that sends ClientJoin and inspects the first server message:
  - MatchInfo → Valid (room exists, host is waiting)
  - Error/RoomNotFound → NotFound (host disconnected / never existed)
  - Error/RoomExpired → Expired (host waited too long)
  - TCP connect failure → NetworkError (relay unreachable / firewall)
  - Bad code format → InvalidCode (wrong length / bad chars)

Closes the probe socket immediately after classification — does NOT
proceed with the full handshake. The full RelayClient handshake opens
a fresh TCP connection after validation succeeds.

== Fix 2: Pre-handshake validation in session.cpp ==
`start_relay_join()` now calls `validate_room_code()` BEFORE setting
up the ENet transport + RelayClient. If validation fails:
  - Sets `room_validation_` field (GUI reads this)
  - Sets descriptive `error_message_` with label + suggestion
  - Returns false immediately (no 60s wait)

If validation succeeds, proceeds with the normal handshake (clears
`room_validation_` so GUI doesn't show stale result).

== Fix 3: Richer relay status messages (session.cpp step_relay) ==
Updated the status strings for each RelayState to be more descriptive:
  - TcpConnecting: shows relay address (zzcaster.duckdns.org:3939)
  - WaitingForHosted: "Registering room with relay..."
  - WaitingForMatchInfo: host vs client phrasing
  - HolePunching: "this can take a few seconds" hint
  - Retrying: shows attempt number

== Fix 4: Enhanced waiting_for_peer.cpp GUI ==
Added two new visual elements:

  1. `draw_relay_phase()` — replaces the single-line status with a
     color-coded phase display:
       - Orange for HolePunching (NAT traversal, may take time)
       - Yellow for Retrying (transient)
       - Red for error/failure phases
       - Muted gray for normal phases

  2. `draw_room_validation_error()` — when `room_validation_` is set
     (room not found / expired / network error), draws a red "ROOM
     CODE ERROR" section with:
       - The specific error label ("Room not found", etc.)
       - A actionable suggestion ("Ask the host to re-create the room",
         "Check firewall TCP port 3939", etc.)

== Files changed ==
  - relay_client.hpp: RoomValidationResult enum + validate_room_code() +
    room_validation_label/suggestion() declarations
  - relay_client.cpp: validate_room_code() implementation (~140 LOC)
  - session.hpp: room_validation_ field + room_validation() getter
  - session.cpp: validation call in start_relay_join + richer status
  - waiting_for_peer.cpp: draw_relay_phase + draw_room_validation_error

== Validation ==
Build passes clean (cmake --build, exit 0). caster.exe recompiled +
linked. The validation probe was tested manually against the relay
server (zzcaster.duckdns.org:3939) — correctly returns NotFound for
nonexistent rooms and Valid for live ones.

== User experience improvement ==
Before: user types #FTTH → "Connecting to relay server..." → 60s wait
        → "No opponent joined (60s timeout)" (misleading! the room
        didn't exist, not a timeout)

After: user types #FTTH → 1-2s validation → immediate red error:
       "ROOM CODE ERROR
        Room not found
        Ask the host to re-create the room and share the new code.
        The host's caster.exe may have disconnected from the relay."

       OR if room is valid → proceeds to handshake with granular
       status: "Connecting to relay server..." → "Joining host's room
       via relay..." → "Negotiating connection details..." → "Hole-
       punching through NAT (this can take a few seconds)..." →
       "Connected via relay!"
