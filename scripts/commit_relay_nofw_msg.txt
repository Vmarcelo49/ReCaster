fix(relay): remove incorrect port-forwarding advice from error messages

The previous commit (2bf98d6) introduced error/suggestion messages
that told users to "open TCP port 3939" or "try port-forwarding" when
the relay connection failed. This is conceptually wrong — relay servers
exist precisely so that players do NOT need to open or forward any
ports.

== What was wrong ==

1. room_validation_suggestion(NetworkError) said:
   "Check your internet connection and firewall (TCP port 3939 must be open)."

   Wrong: TCP 3939 is the signaling channel between caster.exe and the
   relay server. It's an OUTBOUND connection initiated by caster.exe,
   which consumer firewalls/NATs allow by default. No inbound port
   forwarding is needed. Telling the user to open a port defeats the
   entire purpose of using a relay.

2. error_suggestion(HolePunchFailed) said:
   "Your NAT may be too restrictive. Try port-forwarding or a different network."

   Wrong: if UDP hole-punching fails, it usually means one peer is
   behind symmetric NAT (carrier-grade NAT on 4G/LTE, some ISPs). The
   fix is to switch networks, NOT to configure port forwarding — the
   relay was supposed to avoid that requirement.

== What it says now ==

NetworkError:
  "Could not reach the relay server. Check your internet connection
   and try again. No port forwarding is required — the relay handles
   NAT traversal automatically."

HolePunchFailed:
  "Your NAT may be too restrictive (symmetric NAT is common on 4G/LTE).
   Try a different network (home Wi-Fi or wired). No port forwarding
   is required — the relay handles NAT traversal automatically."

Both messages now explicitly state that no port forwarding is needed,
which is the whole point of using a relay server for NAT traversal.

Added inline comments explaining the relay's design philosophy so
future contributors don't reintroduce the same mistake.

  - relay_client.cpp: 2 suggestion strings corrected + explanatory comments

Build passes clean. caster.exe + hook.dll relinked.
