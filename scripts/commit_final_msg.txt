2. FIX(rematch): auto-input select Rematch in RetryMenu
================================================================

== Problem ==
In auto-input mode (CASTER_AUTO_INPUT=1), when a match ended and the
game entered RetryMenu, the auto-input would mash CONFIRM at the
default cursor position (currentMenuIndex=0). MBAACC retry menu layout:
  0 = CharaSelect, 1 = Rematch, 2 = Save Replay

Result: the game fell back to character select instead of rematching.
Both peers captured _localRetryMenuIndex=0, computed
_targetMenuIndex=max(0,0)=0 clamped to min(0,1)=0, and confirmed
index 0 = CharaSelect.

== Root cause ==
Investigation with 3 parallel sub-agents (CCCaster comparison + 5
suspects analysis) confirmed: the auto-input block in dll_main.cpp
generated input inline but never navigated the cursor. The normal
getRetryMenuInput() capture path (menuConfirmState==1) fired and
captured currentMenuIndex=0, but no code forced the index to 1.

== Fix (primary) — dll_main.cpp auto-input block ==
Added a dedicated RetryMenu handler in the auto-input block that calls
g_netMan.setLocalRetryMenuIndex(1) every frame while in RetryMenu
(netplay only). This bypasses the human-driven menuConfirmState
capture and directly selects "Rematch" (index 1).

Flow:
  1. Frame 1 RetryMenu: setLocalRetryMenuIndex(1) -> sendMenuIndex
     fires -> peer receives index 1.
  2. Peer sets _remoteRetryMenuIndex=1.
  3. Both sides have local=1, remote=1 -> getRetryMenuInput computes
     _targetMenuIndex=max(1,1)=1, clamped to min(1,1)=1.
  4. getMenuNavInput() navigates cursor 0->1 (down press), confirms
     at index 1 = Rematch.
  5. Game transitions RetryMenu -> Loading -> CharaIntro -> InGame.

== Fix (secondary) — manager.cpp getRetryMenuInput ==
Re-added the dropped CCCaster branch (DllNetplayManager.cpp:384-397):
  if (currentMenuIndex == 2 ||
      *CC_MENU_STATE_COUNTER_ADDR > _retryMenuStateCounter) {
      menuConfirmState = 2;
      return input;
  }

This branch allows confirms when the player navigated to "Save Replay"
(index 2) or when a submenu opened in front of the retry menu
(CC_MENU_STATE_COUNTER increased). Without it, the netplay branch
unconditionally resets menuConfirmState=0, blocking ALL confirms and
hanging the game if a submenu opens.

The autoReplaySaveStatePtr check from CCCaster is omitted (ReCaster
stripped the saveReplay ASM hook).

== Supporting changes ==
  - manager.hpp: added public setter setLocalRetryMenuIndex(int8_t)
    with documentation explaining it's for auto-input mode.
  - manager.cpp: implemented the setter (idempotent, overwrites
    _localRetryMenuIndex + logs "(forced)" suffix for distinguishability).

3. FIX: code-review bugs (uninit input, rollback loop, B/C swap)
================================================================

Three bugs found during the post-fix code review. Two are pre-existing
(not regressions), one is a long-standing input mapping error.

== B1 (HIGH): uninitialized input in getRetryMenuInput offline path ==
manager.cpp declared `uint16_t input;` without initialization.
The netplay branch assigned it via getRawInput(player), but the
offline branch (else) only set menuConfirmState=2 and never assigned
input. Returning input was UB.

Fix: initialize to 0 and call getRawInput(player) in the offline
branch too, matching CCCaster's behavior (offline passes raw input
through while allowing regular menu operation).

== B2 (HIGH): infinite rollback retry loop on loadState failure ==
dll_main.cpp — if g_rollMan.loadState() returned false, the code only
warned and fell through. The divergence flag (getLastChangedFrame())
was still set and g_rollbackTimer was still at g_minRollbackSpacing,
so the next frame re-triggered the same rollback -> infinite loop
with warn spam.

Fix: on loadState failure, call g_netMan.clearLastChangedFrame()
and --g_rollbackTimer to break the cycle. This mirrors the success
path (which also clears + decrements).

== B/C button swap (user-reported): default Xbox mapping ==
mapping.cpp default_xbox() had buttons B and C swapped relative to
the MBAACC standard diamond layout:
  Correct:  A=X(left)  B=Y(top)    C=B(right)  D=A(bottom)
  Was:      A=X(left)  B=B(right)  C=Y(top)    D=A(bottom)

Fix: swapped the SDL button indices for B (1->3, now Y) and C
(3->1, now B). Updated the color comments in both mapping.cpp
default_xbox() and mapping.hpp field declarations (X is blue, not
green; A is green, not blue).

NOTE: this only affects the DEFAULT mapping. Existing mapping.ini
files saved before this commit retain the old (incorrect) bindings.
Users must either delete caster/mapping.ini or reset to defaults
in the GUI to pick up the fix.
